# Governance

Fallow uses a lightweight maintainer-led model while the project is pre-alpha.

## Roles

- **Contributors** participate through issues, discussions, documentation, code and review.
- **Maintainers** triage, review, merge, release, moderate and set project direction. Current
  maintainers are listed in [`MAINTAINERS.md`](MAINTAINERS.md).

Consistent contributors may be invited to become maintainers based on sustained quality,
judgement, collaboration and alignment with the Code of Conduct. Maintainers may step down at
any time or be removed for prolonged inactivity or conduct violations after private discussion.

## Decisions

Routine changes are decided through pull request review. Significant architectural, protocol,
security, governance or compatibility changes start with a public issue and usually an ADR.
Maintainers seek rough consensus and document unresolved trade-offs. When consensus cannot be
reached in a reasonable time, the lead maintainer makes the decision and records the rationale.

Security incidents and moderation reports may be handled privately. Other project decisions
should be made in public repository channels whenever possible.

## Releases and changes

Maintainers approve releases, follow [`docs/releasing.md`](docs/releasing.md), and apply the
stability rules in [`docs/api-stability.md`](docs/api-stability.md). No contributor may promise
support, compatibility, security certification or legal compliance on behalf of the project.

Governance changes use the same issue and pull request process as code changes and require
approval from a majority of active maintainers.
